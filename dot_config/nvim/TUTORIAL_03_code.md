## Step 1: ファイル内の検索・置換

### 検索

| キー | 動作 |
|------|------|
| `/{検索語}` + `Enter` | 下方向に検索 |
| `?{検索語}` + `Enter` | 上方向に検索 |
| `n` | 次の一致箇所へ |
| `N` | 前の一致箇所へ |
| `*` | カーソル下の単語を検索 |
| `Esc` | 検索ハイライトを消す |

### 置換（Command モード）

```
:%s/old/new/g     → ファイル全体で置換
:%s/old/new/gc    → 確認しながら置換（y=実行 / n=スキップ / q=中止）
:10,20s/old/new/g → 10〜20行目の範囲で置換
```

### フロートウィンドウ内でも検索できる

`<Leader>?u` などで開いたドキュメントのフロートウィンドウ内でも `/` で検索できる。

```
/Step   → 「Step」を検索
n       → 次の「Step」へ
N       → 前の「Step」へ
Esc     → 検索を終了
```

### 練習 1-A

1. ファイルを開いた状態で `/function` + `Enter`
2. `n` で次の一致箇所へ移動
3. `*` でカーソル下の単語を検索
4. `Esc` でハイライトを消す

## Step 4: LSP ジャンプ（定義・参照・実装）

コードの定義元・参照先・実装に飛ぶ。

| キー | 動作 | 飛び先 |
|------|------|--------|
| `gd` | definition | 型定義・宣言（.d.ts や import 元） |
| `gi` | implementation | 実際の実装コード |
| `gr` | references | 使われている全箇所（Telescope で一覧表示） |
| `K` | hover | ドキュメント・型情報をポップアップ表示 |
| `Ctrl+o` | 前の場所へ戻る | ジャンプ前の位置に戻る |

### TypeScript での定義ジャンプ（`gd` の動作）

TypeScript ファイルでの `gd` は `_typescript.goToSourceDefinition` を経由して動作する。

```tsx
// プロジェクト内コンポーネント → .tsx 実装ファイルへ直接ジャンプ
<MyComponent />

// 外部ライブラリ（@c-fo/standard-ui 等）→ SplitPanel.d.ts 等の型定義ファイルへジャンプ
import { SplitPanel } from "@c-fo/standard-ui"
```

内部フロー:

```
gd を押す
  ↓
_typescript.goToSourceDefinition を呼ぶ
  ↓
プロジェクト内コンポーネント → .tsx 実装ファイルへ直接ジャンプ
外部ライブラリ → バンドルファイル（.mjs 等）が返る → 除外
  ↓
textDocument/definition を同じカーソル位置で呼ぶ
  ↓
SplitPanel.d.ts 等の型定義ファイルへジャンプ（VSCode と同じ挙動）
```

> `gi`（implementation）も使える: interface の実装クラスへジャンプする。
> 飛んだ後は `Ctrl+o` で元の場所に戻れる。

### LSP が動いているか確認する

`gd` などが動かない場合はまず LSP が起動しているか確認する。

```
:LspInfo    → 現在のファイルに対して起動している LSP サーバーの一覧
:Mason      → LSP サーバーのインストール状況（✓ = インストール済み）
:LspRestart → LSP サーバーを再起動
:LspLog     → LSP のエラーログを確認
```

#### LSP の仕組み（nvim-lspconfig v2 / Neovim 0.11+）

この設定では `vim.lsp.enable("server_name")` で各サーバーを有効化している。
有効化されたサーバーは対応する filetype のファイルを開いた時に自動起動する。

起動の流れ:
```
.ts ファイルを開く
  → vim.lsp.enable("ts_ls") が呼ばれているため ts_ls が自動起動
  → on_attach が実行され gd / K 等のキーバインドが登録される
```

#### `ts_ls: no active clients` と表示される場合

1. `.tsx` / `.ts` ファイルを開いた状態で `:LspInfo` を確認
2. `tsconfig.json` があるディレクトリ（`front/`）からファイルを開いているか確認
   （`lua/lsp/typescript.lua` の `root_dir` が `tsconfig.json` を起点にルートを検出する）
3. `:Mason` で `ts_ls` が `✓` になっているか確認
4. worktree 環境の場合 `node_modules` が存在するか確認（未インストールなら `npm i`）
5. `:LspRestart` で再起動を試みる

## Step 5: Git 操作

### 変更を確認する（inline）

ファイルを編集すると行番号の左側（ガター）に変更サインが表示される。

| 表示 | 意味 |
|------|------|
| `│`（緑） | 追加行 |
| `│`（黄） | 変更行 |
| `_` / `‾` | 削除行 |

| キー | 動作 |
|------|------|
| `]h` | 次の変更ハンクへ移動 |
| `[h` | 前の変更ハンクへ移動 |
| `<Leader>gp` | 変更内容のプレビュー（変更前後を表示） |
| `<Leader>gd` | 現在ファイルの diff を分割表示（変更前後を並べて確認） |
| `<Leader>gb` | 行の git blame（誰がいつ書いたか） |

### diffview で差分確認・ステージング

複数ファイルの差分をファイルツリー付きで確認しながらステージングできる。

| キー | 動作 |
|------|------|
| `<Leader>gv` | 差分ビューを開く（全変更ファイル一覧 + diff） |
| `<Leader>gh` | 現在ファイルのコミット履歴 |
| `<Leader>gH` | リポジトリ全体のコミット履歴 |

diffview は「ファイルパネル（左）」と「diff ビュー（右）」の 2 ペインで構成される。

ファイルパネル（左）— ステージング操作:

| キー | 動作 |
|------|------|
| `j / k` | ファイルを上下移動 |
| `Enter` / `l` | 選択ファイルの diff を表示 |
| `s` または `-` | 選択ファイルをステージ / アンステージ |
| `S` | 全ファイルをステージ |
| `U` | 全ファイルをアンステージ |
| `X` | 選択ファイルの変更を破棄 |

diff ビュー（右）— 差分確認:

| キー | 動作 |
|------|------|
| `]c / [c` | 次/前の変更ハンクへジャンプ |
| `<Leader>e` | ファイルパネルにフォーカスを移す |
| `g?` | キーバインド一覧を表示 |
| `q` | diffview を閉じる |

> ステージングを diffview で行い、コミット・Push は lazygit（`<Leader>gg`）で行う流れが効率的。

### lazygit でコミット・Push・ブランチ操作

```
<Leader>gg  → lazygit を開く
<Leader>gl  → リポジトリ全体のコミットログ
<Leader>gL  → 現在ファイルのコミットログ
```

lazygit 内のパネル切り替え:

| キー | パネル |
|------|--------|
| `1` | Status（リポジトリ概要） |
| `2` | Files（変更ファイル一覧） |
| `3` | Branches（ブランチ一覧） |
| `4` | Commits（コミット履歴） |
| `5` | Stash（退避リスト） |

Files パネル（ステージ・コミット・Push）:

| キー | 動作 |
|------|------|
| `Space` | ファイルをステージ / アンステージ |
| `a` | 全ファイルをステージ / アンステージ |
| `d` | 選択ファイルの diff を表示 |
| `e` | ファイルをエディタで開く |
| `c` | コミット（メッセージ入力 → Enter） |
| `C` | コミット（エディタで詳細なメッセージを書く） |
| `P` | Push（リモートに送信） |
| `p` | Pull（リモートから取得） |
| `f` | Fetch |
| `A` | 直前のコミットにステージ済みの変更を追加（amend） |

Branches パネル（ブランチ操作）:

| キー | 動作 |
|------|------|
| `Space` | ブランチをチェックアウト |
| `n` | 新規ブランチを作成 |
| `d` | ブランチを削除 |
| `r` | ブランチをリベース |
| `M` | 選択ブランチを現在ブランチにマージ |
| `i` | ブランチ名でフィルタ |

Commits パネル（コミット履歴）:

| キー | 動作 |
|------|------|
| `Space` | コミットをチェックアウト（detached HEAD） |
| `g` | コミットにリセット（ soft/hard を選択） |
| `d` | コミットの diff を表示 |
| `e` | コミットメッセージを編集（rebase -i） |
| `s` | コミットを squash（直前にまとめる） |
| `f` | コミットをフィクスアップ（メッセージなしで squash） |
| `p` | Cherry pick |
| `r` | コミットをリバート |

全パネル共通:

| キー | 動作 |
|------|------|
| `?` | ヘルプ（キーバインド一覧） |
| `Esc` | 現在のパネル / モードをキャンセル |
| `q` | lazygit を閉じて Neovim に戻る |

## Step 6: GitHub PR を Neovim で確認・操作（octo.nvim）

`octo.nvim` を使うと GitHub のPR・Issue を Neovim 内で閲覧・コメント投稿できる。

事前条件: `gh` CLI のログイン済み状態が必要（`gh auth login`）。

### PR を開く

| キー | 動作 |
|------|------|
| `<Leader>po` | 現在ブランチの PR を開く |
| `<Leader>pp` | PR 一覧を表示（Telescope で絞り込み） |
| `<Leader>pf` | PR の変更ファイル一覧 |
| `<Leader>pc` | PR の CI チェック状態 |
| `<Leader>pr` | PR レビューを開始 |

### Octo バッファ内の操作

PR を開いた後、専用バッファで以下の操作ができる:

| キー | 動作 |
|------|------|
| `<Leader>ca` | コメントを追加 |
| `<Leader>cb` | コメントを削除 |
| `<Tab>` | 次のコメントへ移動 |
| `<S-Tab>` | 前のコメントへ移動 |
| `<Leader>ic` | PR をクローズ |
| `<Leader>io` | PR を再オープン |
| `<Leader>im` | PR をマージ |
| `<Leader>la` | ラベルを追加 |
| `<Leader>raa` | レビュワーを追加 |

### Octo コマンド一覧

```
:Octo pr list             → PR 一覧
:Octo pr edit             → 現在ブランチの PR を開く
:Octo pr files            → 変更ファイル一覧
:Octo pr checks           → CI チェック状態
:Octo pr merge            → PR をマージ
:Octo review start        → レビュー開始
:Octo review submit       → レビュー提出
:Octo issue list          → Issue 一覧
:Octo issue create        → Issue 作成
```

### 練習 6-A: PRを確認する

1. GitHub にPRがあるブランチで Neovim を開く
2. `<Leader>po` で現在ブランチのPRを開く
3. PR 本文・コメント・変更ファイルを確認
4. `<Leader>pf` で変更ファイル一覧を表示
5. `q` でバッファを閉じる

### 練習 5-A: gitsigns で inline 確認

1. ファイルを適当に編集して保存（`:w`）
2. ガターに `│` が表示されることを確認
3. `]h` で変更ハンクへ移動
4. `<Leader>gp` で変更前後の差分を確認

### 練習 5-B: diffview でステージング

1. `<Leader>gv` で差分ビューを開く
2. ファイルパネル（左）で `j/k` を使いファイルを確認
3. `l` で選択ファイルの diff を diff ビュー（右）に表示
4. `]c / [c` で diff ビュー内のハンクを移動
5. ファイルパネルに戻り `s` でステージ
6. `S` で残りの全ファイルをステージ
7. `q` で diffview を閉じる

### 練習 5-C: lazygit でコミット

1. `<Leader>gg` で lazygit を開く
2. `Space` でファイルをステージ（diffview でステージ済みの場合は確認のみ）
3. `d` で diff を確認
4. `c` でコミット（メッセージ入力 → Enter）
5. `q` で lazygit を閉じて Neovim に戻る
